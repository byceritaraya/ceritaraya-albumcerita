## 1. Overview

AlbumCerita is a premium collaborative event photo album platform under the Cerita Raya ecosystem.

Guests scan a QR code, enter their name, take photos, and contribute memories into a shared event album.

The product focuses on memories, stories, and shared moments — not utility.

This is not a disposable camera clone.

## 5. Album Lifecycle

An event progresses through the following states:

```
Draft → Published → Expired → Archived
```

| State | Description | Guest Can Upload | Guest Can View Gallery |
|-------|-------------|:---------------:|:--------------------:|
| **Draft** | Event is live. Uploads are open. Gallery is private. | ✅ Yes | ❌ No |
| **Published** | Album is final. Uploads are permanently closed. Gallery is public. | ❌ No | ✅ Yes |
| **Expired** | Retention period has passed. Event is read-only. | ❌ No | ❌ No |
| **Archived** | Data is archived or deleted per retention policy. | ❌ No | ❌ No |

**Default state on creation:** `Draft`

**Transition rules:**

| Trigger | From | To | Who |
|---------|------|----|-----|
| Admin creates event | — | `Draft` | Admin |
| Client clicks "Publish Album" | `Draft` | `Published` | Client |
| `expires_at` date passes (automated) | `Published` | `Expired` | System |
| Admin archives event | `Expired` | `Archived` | Admin |
| Admin reverts album | `Published` | `Draft` | Admin only |

**Publishing an album means:**
- Client has finished reviewing photos
- Album is considered final
- **Guest uploads are permanently disabled**
- **Public gallery becomes visible to all guests**

> "Publish Album" acts as **Lock Uploads + Open Gallery** in a single action.

> Clients cannot revert a Published album back to Draft. Only admin can.

## 7. Guest Experience

### 7.1 Entry Flow

```
Scan QR Code
→ https://ceritaraya.com/albumcerita/[slug]
→ Enter Guest Name
→ Camera Access (if state = Draft AND contributor limit not reached)
→ Gallery (if state = Published)
```

1. Guest scans QR code at the event venue.
2. Guest lands on the event page at `/albumcerita/[slug]`.
3. Guest is prompted to enter their **first name** before accessing the camera.
4. A random UUID `guest_token` is generated and stored in `localStorage` on first visit.
5. Guest name is stored in `localStorage` alongside `guest_token`. Re-used on subsequent visits from the same browser.
6. Camera access is gated by: album state must be `Draft`, contributor limit must not be exceeded for **new** contributors.

### 7.2 Guest Name Requirements

- Required field. Cannot be skipped.
- First name only (no last name required).
- Minimum 1 character, maximum 50 characters.
- Displayed in the gallery alongside the guest's photos.
- Stored in `localStorage` on the guest's device. Re-used on subsequent visits from the same browser.
- Not stored as a user account. Not tied to any login.

### 7.3 Guest Token

- Generated as a **random UUID** (e.g. `uuid.v4()`) on first visit to any event page.
- Stored in `localStorage` under the key `ac_guest_token`.
- Persisted across sessions on the same browser.
- Used server-side to track per-guest photo count and contributor status.
- **Not** derived from device hardware, IP address, or any fingerprinting technique.
- Not considered personally identifiable information.

### 7.4 Photo Limits

- Each event configures a **photos per guest** limit.
- Limit is enforced **server-side** (by `guest_token` + `event_id`).

| Option | Value |
|--------|-------|
| Option A | 5 photos per guest |
| Option B | 10 photos per guest |
| Option C | 20 photos per guest |
| Option D | 36 photos per guest |

- When the guest reaches their limit, the camera is disabled and a completion screen is shown.
- Remaining photo count is displayed during the camera experience.

### 7.5 Contributor Limit

Each event configures a **max_contributors** limit — the maximum number of distinct guests who may upload photos.

| Option | DB Value |
|--------|----------|
| 20 Contributors | `20` |
| 50 Contributors | `50` |
| 100 Contributors | `100` |
| Unlimited | `NULL` |

**Definition:** A contributor is counted only when they successfully upload their **first photo** to the event.

- Opening the event page does **not** consume a contributor slot.
- Browsing the gallery does **not** consume a contributor slot.
- A contributor slot is **permanently assigned** to that `guest_token` for the duration of the event, even if their photos are later trashed.

**Limit enforcement logic (server-side, at upload time):**

```
Is album state = Draft?        → No → Reject upload (uploads closed)
Does guest_token already have a photo in this event?
  → Yes → Allow upload (existing contributor, slot already used)
  → No  → Is contributor count < max_contributors (or max_contributors IS NULL)?
             → Yes → Allow upload (new contributor, slot assigned)
             → No  → Reject with contributor limit message
```

**When contributor limit is reached:**

- New guests may still access the event page.
- New guests may still browse the gallery (if Published).
- If a new guest attempts to upload their first photo, show:

> *"Album ini telah mencapai jumlah kontributor maksimum yang ditentukan penyelenggara."*

- Existing contributors (whose `guest_token` already has a photo) can continue uploading up to their per-guest photo limit.

### 7.6 Guest Gallery Visibility

- When album state is **Draft**: guests see no gallery. A message informs them the album will be available after the event.
- When album state is **Published**: guests can view the full gallery, including photos taken by all other guests.

### 7.7 Guest Event Page Features

| Feature | Notes |
|---------|-------|
| Event Cover Photo | Full-width hero image |
| Event Name & Date | Displayed on cover |
| Venue Name | Displayed on cover |
| Welcome Message | Editable by admin, shown to guests |
| Guest Name Entry | Required before camera access |
| Camera Button | Shown when state = Draft AND guest is eligible to upload |
| Gallery | Shown only when state = Published |
| Photo Count | Live count (real-time via Supabase Realtime) |

## 8. Camera Experience

### 8.1 Requirements

- Fullscreen camera view. No navigation bars.
- Default camera: **Rear (back-facing)**
- Front/rear camera toggle
- Large capture button
- Remaining photo counter (e.g., "8 foto tersisa")
- Last captured photo thumbnail (bottom-left)
- Uses browser MediaDevices API — no native app required

### 8.2 Permission Flow

Before triggering the browser camera permission dialog, show a **permission priming screen**:

> *"AlbumCerita butuh akses kamera untuk mengambil foto dan menambahkannya ke album bersama."*
> [Izinkan Akses Kamera] [Tidak Sekarang]

Only request permission after the guest taps "Izinkan".

### 8.3 Upload Behaviour

- Photo is captured and immediately shown as a thumbnail.
- Upload begins immediately in the background — non-blocking.
- Guest can take another photo immediately without waiting.
- Upload progress shown on thumbnail (ring indicator).
- On success: toast notification — *"Foto berhasil ditambahkan ke album"*
- On failure: error message shown with a **Retry** button — *"Gagal mengunggah."* + [Coba Lagi]
- Guest manually taps Retry to attempt re-upload.
- **No automatic retry. No offline queue. No service workers.**

### 8.4 Camera States

| State | Trigger | Screen Shown |
|-------|---------|-------------|
| **Permission Priming** | Camera not yet permitted | Priming screen (§8.2) |
| **Active** | Permission granted, album is Draft, guest is eligible | Live viewfinder |
| **Uploading** | Upload in progress | Viewfinder + thumbnail ring indicator |
| **Upload Failed** | Upload error | Error toast + Retry button |
| **Per-Guest Limit Reached** | Guest has used all their allowed photos | Completion screen (§8.5) |
| **Contributor Limit Reached** | New guest, event contributor limit full | Contributor limit screen (§8.6) |
| **Uploads Closed** | Album state is Published, Expired, or Archived | Uploads closed screen (§8.7) |
| **Unsupported** | Browser does not support MediaDevices API | Unsupported screen (§8.8) |

### 8.5 Per-Guest Limit Reached Screen

> *"Terima kasih! Semua fotomu sudah masuk ke album."*
> [Lihat Album]

### 8.6 Contributor Limit Reached Screen

> *"Album ini telah mencapai jumlah kontributor maksimum yang ditentukan penyelenggara."*
> [Lihat Album]

### 8.7 Uploads Closed Screen

Shown when a guest attempts to access the camera after the album has been Published, Expired, or Archived:

> *"Album ini sudah ditutup. Foto tidak lagi dapat ditambahkan."*
> [Lihat Album]

### 8.8 Unsupported Browser Screen

> *"Browser Anda tidak mendukung kamera. Silakan gunakan browser lain atau upload foto dari galeri."*

| Field | Format |

```

→ Enter Event ID + PIN
→ Server validates credentials
→ Create session cookie
→ Redirect to /albumcerita/manage/[eventId]

```

- Validation is server-side only.
- On success: session cookie created → redirect to `/albumcerita/manage/[eventId]`
- On failure: *"Event ID atau PIN tidak valid."* (do not specify which field is wrong)
- Rate limit: 5 failed attempts → 15-minute lockout per IP

No registration. No username. No password.

### 9.2 Client Dashboard

Route: `/albumcerita/manage/[eventId]`

Access: PIN session cookie (short-lived, expires after 24 hours of inactivity)

#### Stats Cards (top of dashboard)

| Stat | Description |
|------|-------------|
| Total Photos | Count of all uploaded photos excluding trashed (`deleted_at IS NULL`) |
| Contributors | `[used] / [limit]` — e.g. "18 / 50 Contributors". Shows "Unlimited" if `max_contributors IS NULL`. |
| Album State | Current lifecycle state badge (`Draft`, `Published`, `Expired`, `Archived`) |
| Expires On | Date formatted as "15 Desember 2026" |

#### Actions

| Action | Available When | Description |
|--------|---------------|-------------|
| **Publish Album** | State = Draft | Changes state to `Published`. Locks uploads. Opens public gallery. Requires confirmation dialog. Irreversible by client. |
| **Download All Photos** | Any state | Triggers async ZIP generation. Includes visible, non-trashed photos only. Client receives download link when ready. |
| **Copy Guest Link** | Any state | Copies `https://ceritaraya.com/albumcerita/[slug]` to clipboard. Shows toast. |
| **Download QR Code** | Any state | Downloads print-ready PNG (300 DPI, min 1000×1000px) of the event QR code. |

**Publish Album confirmation dialog:**

> *"Setelah dipublish, tamu tidak dapat lagi menambahkan foto. Album akan menjadi publik dan tidak bisa dikembalikan ke Draft. Lanjutkan?"*
> [Batalkan] [Ya, Publish Album]

#### Photo Moderation

Clients can moderate photos in their own event only:

| Action | Description | Reversible? |
|--------|-------------|:-----------:|
| **Hide Photo** | Hides photo from public gallery. Sets `is_hidden = true`. Photo remains in all client views. | ✅ Client can unhide |
| **Move to Trash** | Sets `deleted_at = now()`. Removed from main gallery and stats. Not permanently deleted. | ✅ Client can restore |
| **Restore from Trash** | Clears `deleted_at`. Photo returns to main gallery. | ✅ |

> **Clients cannot permanently delete photos.**
> Permanent deletion is admin-only.
> Trashed photos remain in storage and database until an admin permanently deletes them.

#### Trash View

- Accessible via a "Sampah" (Trash) tab or section within the client dashboard.
- Displays all photos where `deleted_at IS NOT NULL` for this event.
- Each photo shows guest name, upload time, and trash date.
- Available action per photo: **Restore from Trash**.
- No bulk permanent delete available to clients.

#### Gallery View

- Masonry grid layout.
- When album is Draft: all non-trashed photos visible to client (for review before publishing).
- Trashed photos (`deleted_at IS NOT NULL`) are not shown in the main gallery — only in the Trash view.
- Hidden photos appear dimmed with a "Disembunyikan" label — visible to client only, not to guests.
- Each photo displays the guest name.
- Tapping a photo opens a fullscreen lightbox with guest name and upload timestamp.
- Lightbox actions: Hide / Move to Trash (or Unhide if currently hidden).

### Key Indexes

```sql
CREATE INDEX idx_photos_event_id ON photos(event_id);
CREATE INDEX idx_photos_event_id_uploaded_at ON photos(event_id, uploaded_at DESC);
CREATE INDEX idx_photos_guest_token ON photos(event_id, guest_token);
CREATE INDEX idx_photos_deleted_at ON photos(event_id, deleted_at);
CREATE INDEX idx_events_slug ON events(slug);
CREATE INDEX idx_events_event_id ON events(event_id);
CREATE INDEX idx_pin_attempts_ip ON pin_attempts(ip_address, attempted_at);
```

### Common Query Patterns

```sql
-- Active photos for public gallery (Published album)
SELECT * FROM photos
WHERE event_id = $1
  AND is_hidden = false
  AND deleted_at IS NULL
ORDER BY uploaded_at DESC;

-- All photos for client gallery review (including hidden, excluding trashed)
SELECT * FROM photos
WHERE event_id = $1
  AND deleted_at IS NULL
ORDER BY uploaded_at DESC;

-- Trashed photos for client Trash view
SELECT * FROM photos
WHERE event_id = $1
  AND deleted_at IS NOT NULL
ORDER BY deleted_at DESC;

-- Per-guest photo count for limit enforcement
SELECT COUNT(*) FROM photos
WHERE event_id = $1
  AND guest_token = $2
  AND deleted_at IS NULL;

-- Contributor count for an event
-- NOTE: counts ALL distinct tokens ever uploaded, regardless of trash status.
-- A contributor slot is permanent once the first photo is uploaded.
SELECT COUNT(DISTINCT guest_token) FROM photos
WHERE event_id = $1;

-- Check if a specific guest_token is an existing contributor
SELECT EXISTS (
  SELECT 1 FROM photos
  WHERE event_id = $1
    AND guest_token = $2
);
```

## 16. Security Requirements

| Requirement | Specification |
|------------|---------------|
| HTTPS | Required. Camera MediaDevices API will not function on HTTP. |
| PIN storage | Stored as bcrypt hash. Never stored in plaintext. |
| Rate limiting | PIN login: 5 attempts per 15 minutes per IP |
| Lockout | After 5 failed PIN attempts: 15-minute IP lockout |
| Upload validation | Server-side MIME type check. Accept: `image/jpeg`, `image/png`, `image/webp` only. Max file size: 15MB. |
| Upload state check | Server must validate `event.state = 'draft'` before accepting any upload. |
| Contributor check | Server must validate contributor limit server-side before accepting any first upload from a new guest_token. |
| EXIF stripping | Strip all EXIF metadata from uploaded photos server-side before storage |
| Admin auth | Supabase Auth (email + password) with JWT sessions |
| Session management | Client sessions expire after 24 hours of inactivity |
| Event deactivation | Admin can disable uploads on an event without changing its state |
| Permanent deletion | Restricted to admin role only. Client actions are non-destructive (Trash only). |

## 19. Success Criteria

The MVP is successful when:

1. A guest can scan a QR code, enter their name, take a photo, and have it appear in the album — in under 60 seconds, on a mobile browser, without installing an app.
2. A client can log in at `/albumcerita/event` with their Event ID and PIN, review their photos, and publish their album without requiring help from Cerita Raya.
3. An admin can create a complete, ready-to-use event (with QR code) in under 5 minutes.
4. The product feels premium — not like a SaaS tool.

### The One Test

Show the product to someone unfamiliar with it.

They should say: **"Ini cantik. Gimana cara pakainya?"**

Not: **"Ini dashboard-nya di mana?"**
