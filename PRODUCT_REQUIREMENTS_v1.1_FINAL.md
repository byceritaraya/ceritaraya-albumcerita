## 1. Overview

AlbumCerita is a premium collaborative event photo album platform under the Cerita Raya ecosystem.

Guests scan a QR code, enter their name, take photos, and contribute memories into a shared event album.

The product focuses on memories, stories, and shared moments — not utility.

This is not a disposable camera clone.

## 5. Album Lifecycle

An event progresses through the following states:

```
Draft → Published → Expired → Archived
```

| State | Description | Guest Can Upload | Guest Can View Gallery |
|-------|-------------|-----------------|----------------------|
| **Draft** | Event is live. Guests can upload photos. Gallery is private. | ✅ Yes | ❌ No |
| **Published** | Client has reviewed and published the album. | ✅ Yes | ✅ Yes |
| **Expired** | Retention period has passed. Event is read-only. | ❌ No | ❌ No |
| **Archived** | Data is archived or deleted per retention policy. | ❌ No | ❌ No |

**Default state on creation:** Draft

**Transition rules:**
- Admin creates event → state is `Draft`
- Client clicks "Publish Album" → state changes to `Published`
- `expires_at` date passes → state changes to `Expired` (automated)
- Admin archives expired event → state changes to `Archived`

> Clients cannot revert a Published album back to Draft. Only admin can.

## 7. Guest Experience

### 7.1 Entry Flow

```
Scan QR Code
→ https://ceritaraya.com/albumcerita/[slug]
→ Enter Guest Name
→ Camera Access
→ (Gallery visible only if Published)
```

1. Guest scans QR code at the event venue.
2. Guest lands on the event page at `/albumcerita/[slug]`.
3. Guest is prompted to enter their **first name** before accessing the camera.
4. Name is stored in `localStorage` — no account, no registration.
5. Guest accesses the camera and takes photos.

### 7.2 Guest Name Requirements

- Required field. Cannot be skipped.
- First name only (no last name required).
- Minimum 1 character, maximum 50 characters.
- Displayed in the gallery alongside the guest's photos.
- Stored in `localStorage` on the guest's device. Re-used on subsequent visits from the same browser.
- Not stored as a user account. Not tied to any login.

### 7.3 Photo Limits

- Each event configures a **photos per guest** limit.
- Limit is enforced **server-side** (by guest token + event ID).

| Option | Value |
|--------|-------|
| Option A | 5 photos per guest |
| Option B | 10 photos per guest |
| Option C | 20 photos per guest |
| Option D | 36 photos per guest |

- When the guest reaches their limit, the camera is disabled and a completion screen is shown.
- Remaining photo count is displayed during the camera experience.

### 7.4 Guest Gallery Visibility

- When album state is **Draft**: guests see no gallery. A message informs them the album will be available after the event.
- When album state is **Published**: guests can view the full gallery, including photos taken by all other guests.

### 7.5 Guest Event Page Features

| Feature | Notes |
|---------|-------|
| Event Cover Photo | Full-width hero image |
| Event Name & Date | Displayed on cover |
| Venue Name | Displayed on cover |
| Welcome Message | Editable by admin, shown to guests |
| Guest Name Entry | Required before camera access |
| Camera Button | Primary CTA — opens camera |
| Gallery | Shown only when state is Published |
| Photo Count | Live count (real-time via Supabase Realtime) |

## 8. Camera Experience

### 8.1 Requirements

- Fullscreen camera view. No navigation bars.
- Default camera: **Rear (back-facing)**
- Front/rear camera toggle
- Large capture button
- Remaining photo counter (e.g., "8 foto tersisa")
- Last captured photo thumbnail (bottom-left)
- Uses browser MediaDevices API — no native app required

### 8.2 Permission Flow

Before triggering the browser camera permission dialog, show a **permission priming screen**:

> *"AlbumCerita butuh akses kamera untuk mengambil foto dan menambahkannya ke album bersama."*
> [Izinkan Akses Kamera] [Tidak Sekarang]

Only request permission after the guest taps "Izinkan".

### 8.3 Upload Behaviour

- Photo is captured and immediately shown as a thumbnail.
- Upload begins immediately in the background — non-blocking.
- Guest can take another photo immediately without waiting.
- Upload progress shown on thumbnail (ring indicator).
- On success: toast notification — *"Foto berhasil ditambahkan ke album"*
- On failure: error message shown with a **Retry** button — *"Gagal mengunggah."* + [Coba Lagi]
- Guest manually taps Retry to attempt re-upload.
- **No automatic retry. No offline queue. No service workers.**

### 8.4 Camera States

| State | Description |
|-------|-------------|
| Permission Priming | Before native permission dialog |
| Active | Live viewfinder |
| Uploading | Upload in progress (non-blocking) |
| Upload Failed | Error shown, Retry button displayed |
| Limit Reached | Guest has used all their allowed photos |
| Unsupported | Browser/device does not support camera API |

### 8.5 Limit Reached Screen

When a guest has used all their allowed photos:

> *"Terima kasih! Semua fotomu sudah masuk ke album."*
> [Lihat Album]

### 8.6 Unsupported Browser Screen

When the browser does not support the MediaDevices API:

> *"Browser Anda tidak mendukung kamera. Silakan gunakan browser lain atau upload foto dari galeri."*

## 9. Client Access

### 9.1 Login

Route: `/albumcerita/event`

| Field | Format |
|-------|--------|
| Event ID | `CRA-XXXXXX` — auto-uppercase input |
| PIN | 6-digit numeric — triggers numeric keyboard on mobile |

Button: **Lihat Album**

**Flow:**
```
/albumcerita/event
→ Enter Event ID + PIN
→ Server validates credentials
→ Create session cookie
→ Redirect to /albumcerita/manage/[eventId]

Example: /albumcerita/manage/CRA-7K9X2P
```

- Validation is server-side only.
- On success: session cookie created → redirect to `/albumcerita/manage/[eventId]`
- On failure: *"Event ID atau PIN tidak valid."* (do not specify which field is wrong)
- Rate limit: 5 failed attempts → 15-minute lockout per IP

No registration. No username. No password.

### 9.2 Client Dashboard

Route: `/albumcerita/manage/[eventId]`

Access: PIN session cookie (short-lived, expires after 24 hours of inactivity)

#### Stats Cards (top of dashboard)

| Stat | Description |
|------|-------------|
| Total Photos | Count of all uploaded (non-deleted) photos |
| Total Guests | Count of distinct guest tokens |
| Album State | Current lifecycle state badge |
| Expires On | Date formatted as "15 Desember 2026" |

#### Actions

| Action | Description |
|--------|-------------|
| **Publish Album** | Changes state from Draft → Published. Guests can now view gallery. Requires confirmation dialog. |
| **Download All Photos** | Triggers async ZIP generation. Client receives download link when ready. |
| **Copy Guest Link** | Copies `https://ceritaraya.com/albumcerita/[slug]` to clipboard. Shows toast confirmation. |
| **Download QR Code** | Downloads print-ready PNG (300 DPI, min 1000×1000px) of the event QR code. |

#### Photo Moderation

Clients can moderate photos in their own event only:

| Action | Description |
|--------|-------------|
| **Hide Photo** | Photo is hidden from the public gallery. Remains in database. Reversible by client. |
| **Delete Photo** | Permanently deletes photo from storage and database. Irreversible. Requires confirmation dialog. |

> Clients can only moderate their own event's photos.
> Clients cannot access or modify other events.
> Only admins can restore deleted photos (if a backup exists).

#### Gallery View

- Masonry grid layout.
- When album is Draft: all uploaded photos visible to client (for review before publishing).
- Hidden photos appear dimmed with a "Disembunyikan" label — visible to client only, not to guests.
- Deleted photos are removed entirely from view.
- Each photo displays the guest name.
- Tapping a photo opens a fullscreen lightbox with guest name and upload timestamp.

### 10.2 Event Creation Fields

| Field | Type | Required | Notes |

| Event Name | Text | ✅ | e.g. "Yuda & Anggi Wedding" |

- Admin panel generates QR code for each event.

- QR code displayed in the event detail card alongside the Event ID and PIN.

- Admin can view all photos across all events.
- Admin can hide, delete, or restore (if soft-deleted) any photo.
- Admin can delete entire event galleries.

### 10.5 Admin UI Constraints

- Use card layouts — no tables, no spreadsheets.
- Avoid enterprise SaaS dashboard aesthetics.
- Consistent with the product design system.

## 16. Security Requirements

| Requirement | Specification |
|------------|---------------|
| HTTPS | Required. Camera MediaDevices API will not function on HTTP. |
| PIN storage | Stored as bcrypt hash. Never stored in plaintext. |
| Rate limiting | PIN login: 5 attempts per 15 minutes per IP |
| Lockout | After 5 failed PIN attempts: 15-minute IP lockout |
| Upload validation | Server-side MIME type check. Accept: `image/jpeg`, `image/png`, `image/webp` only. Max file size: 15MB. |
| EXIF stripping | Strip all EXIF metadata from uploaded photos server-side before storage |
| Admin auth | Supabase Auth (email + password) with JWT sessions |
| Session management | Client sessions expire after 24 hours of inactivity |
| Event deactivation | Admin can disable uploads on an event without changing its state |
| Soft delete | Photos are soft-deleted (`is_deleted = true`) before permanent removal |

## 19. Success Criteria

The MVP is successful when:

1. A guest can scan a QR code, enter their name, take a photo, and have it appear in the album — in under 60 seconds, on a mobile browser, without installing an app.
2. A client can log in at `/albumcerita/event` with their Event ID and PIN, review their photos, and publish their album without requiring help from Cerita Raya.
3. An admin can create a complete, ready-to-use event (with QR code) in under 5 minutes.
4. The product feels premium — not like a SaaS tool.

### The One Test

Show the product to someone unfamiliar with it.

They should say: **"Ini cantik. Gimana cara pakainya?"**

Not: **"Ini dashboard-nya di mana?"**
